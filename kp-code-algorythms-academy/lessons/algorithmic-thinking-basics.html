Cele lekcji

Po tej lekcji potrafisz zatrzymać się przed kodowaniem i świadomie zaplanować rozwiązanie.

  • Opisać problem własnymi słowami.
  • Określić dane wejściowe i oczekiwany wynik.
  • Znaleźć ograniczenia, założenia i przypadki brzegowe.
  • Rozbić problem na mniejsze kroki oraz zapisać pseudokod.
  • Przetestować rozwiązanie na prostych i nietypowych przykładach.

Czym jest myślenie algorytmiczne?

Algorithmic thinking, czyli myślenie algorytmiczne, to umiejętność zamiany niejasnego problemu w uporządkowaną sekwencję decyzji i kroków. Algorytm nie musi oznaczać skomplikowanego kodu. To po prostu droga od danych wejściowych do oczekiwanego wyniku.

Dobry algorytm zaczyna się od pytania: „co dokładnie mam otrzymać, co mam zwrócić i po czym poznam, że wynik jest poprawny?”.

Najpierw problem, potem kod

Początkujący często zaczynają od zdania: „napiszę pętlę i zobaczę, co wyjdzie”. To prowadzi do zmieniania rozwiązania w trakcie, niejasnego typu zwracanej wartości i testowania tylko szczęśliwej ścieżki.

Bardziej profesjonalne podejście brzmi: „najpierw określę wejście, wynik, zasady, przypadki brzegowe i dopiero wtedy zapiszę kod”. Kod jest ostatnim etapem planu, a nie sposobem zgadywania wymagań.

Analogia: szukanie produktu na liście zakupów

  1. Punkt początkowy: masz listę produktów i nazwę produktu do znalezienia.
  2. Dostępne informacje: widzisz nazwy, ale nie wiesz z góry, na której pozycji jest produkt.
  3. Kolejne kroki: sprawdzasz pozycje po kolei i porównujesz nazwy.
  4. Warunek zakończenia: kończysz, gdy znajdziesz produkt albo dojdziesz do końca listy.
  5. Rezultat: wskazujesz znaleziony produkt albo mówisz, że nie ma go na liście.

Powtarzalny mental model

1. Understand

Opisz problem prostymi słowami.

2. Inputs / Outputs

Ustal, co funkcja dostaje i co ma zwrócić.

3. Constraints

Zapisz zasady, ograniczenia i założenia.

4. Break down

Podziel rozwiązanie na małe, możliwe do sprawdzenia kroki.

5. Edge cases

Sprawdź puste, nietypowe lub graniczne dane.

6. Plan → Implement → Verify

Najpierw pseudokod, potem JavaScript, na końcu testy przykładów.

Główny przykład: wyszukiwanie użytkownika

Wymaganie: napisz funkcję, która otrzymuje listę użytkowników i nazwę użytkownika, a następnie zwraca znaleziony obiekt użytkownika albo null.

Input

const users = [
  { id: 1, name: "Anna" },
  { id: 2, name: "Marek" },
  { id: 3, name: "Ola" }
];

Funkcja otrzymuje tablicę użytkowników oraz nazwę do wyszukania.

Output

Funkcja zwraca obiekt znalezionego użytkownika albo null, jeśli użytkownik nie istnieje.

Ograniczenia i założenia

  • Czy porównanie rozróżnia wielkość liter: "Marek" kontra "marek"?
  • Czy nazwy mogą się powtarzać i czy zwracamy pierwszy wynik?
  • Co zwrócić dla pustej tablicy lub pustego zapytania?
  • Czy każdy element tablicy ma poprawne pola id i name?
  • Czy oczekiwany wynik to obiekt, indeks, pierwszy wynik czy wszystkie wyniki?

Plan rozwiązania krok po kroku

  1. Przyjmij listę użytkowników i szukaną nazwę.
  2. Ustal, że szukamy dokładnego dopasowania nazwy i zwracamy pierwszy pasujący obiekt.
  3. Przejdź przez użytkowników po kolei.
  4. Porównaj nazwę bieżącego użytkownika z wyszukiwaną nazwą.
  5. Zwróć użytkownika po znalezieniu dopasowania.
  6. Zwróć null, jeśli lista została sprawdzona bez wyniku.

Walkthrough dla konkretnego wywołania

findUserByName(users, "Marek");
  1. Funkcja sprawdza użytkownika "Anna". To nie jest "Marek", więc idzie dalej.
  2. Funkcja sprawdza użytkownika "Marek". Nazwa pasuje.
  3. Funkcja kończy działanie i zwraca obiekt { id: 2, name: "Marek" }.
  4. Użytkownik "Ola" nie musi być sprawdzany, bo wynik został już znaleziony.

Pseudokod

FUNCTION find user by name
  FOR each user in users
    IF user name matches searched name
      RETURN user
  RETURN null
END FUNCTION

Pseudokod celowo omija szczegóły składni JavaScript. Dzięki temu najpierw widzisz logikę, a dopiero potem zapis w konkretnym języku.

Implementacja w JavaScript

function findUserByName(users, searchedName) {
  for (const user of users) {
    if (user.name === searchedName) {
      return user;
    }
  }

  return null;
}

users to lista do przeszukania, a searchedName to wartość, której szukamy. Pętla for...of sprawdza użytkowników po kolei. Warunek user.name === searchedName oznacza dokładne dopasowanie. return user kończy funkcję po znalezieniu pierwszego wyniku, a return null jasno komunikuje brak dopasowania.

Lepsze wymagania zmieniają implementację

Jeśli produkt wymaga wyszukiwania bez rozróżniania wielkości liter, trzeba powiedzieć to przed kodowaniem. Wtedy świadomie normalizujemy obie wartości:

function findUserByNameCaseInsensitive(users, searchedName) {
  const normalizedSearch = searchedName.toLowerCase();

  for (const user of users) {
    if (user.name.toLowerCase() === normalizedSearch) {
      return user;
    }
  }

  return null;
}

To nie jest „lepsza” wersja w każdej sytuacji. To wersja zgodna z innym wymaganiem: lepiej zdefiniowane wymagania → bardziej świadoma implementacja.

Krótkie wprowadzenie do kosztu rozwiązania

W najgorszym przypadku funkcja sprawdzi każdego użytkownika, bo szukana osoba może być na końcu listy albo nie istnieć. Dla n użytkowników czas działania zapisujemy jako O(n). Dodatkowa pamięć to O(1), ponieważ używamy tylko kilku zmiennych pomocniczych. Dokładniejsze wyjaśnienie tej notacji pojawi się w lekcji Big O Notation.

Najczęstsze błędy

  • Pisanie kodu przed zrozumieniem problemu.
  • Brak jasnej decyzji, czy funkcja zwraca obiekt, indeks, null czy tablicę wyników.
  • Ignorowanie pustej tablicy, pustego zapytania i duplikatów.
  • Testowanie tylko jednego poprawnego przypadku.
  • Tworzenie zbyt skomplikowanego rozwiązania dla prostego wymagania.
  • Mieszanie planowania, implementacji i testowania w jednym kroku.
  • Brak decyzji dotyczącej wielkości liter.

Before you code — praktyczna checklista

  • Co otrzymuję?
  • Co mam zwrócić?
  • Jak wygląda poprawny wynik?
  • Jakie dane mogą być niepoprawne lub puste?
  • Jakie są ograniczenia?
  • Czy istnieją duplikaty?
  • Czy kolejność danych ma znaczenie?
  • Jak podzielę rozwiązanie na kroki?
  • Jakie przykłady testowe przygotuję?
  • Skąd będę wiedzieć, że rozwiązanie działa?

Pytania praktyczne i mini challenge

  1. Określ input i output funkcji sprawdzającej, czy produkt jest dostępny.
  2. Wypisz przypadki brzegowe funkcji obliczającej średnią ocen.
  3. Rozbij na kroki funkcję znajdującą najdłuższe słowo w tablicy.
  4. Zdecyduj, co powinna zwrócić funkcja wyszukująca nieistniejący element.

Mini challenge

Zaplanuj funkcję findProductById(products, productId). Zanim napiszesz implementację, zapisz: input, output, założenia, przypadki brzegowe i pseudokod. Dopiero potem przejdź do JavaScriptu.

Podsumowanie

Kod jest ostatnim etapem procesu. Dobre rozwiązanie zaczyna się od zrozumienia problemu, a input, output, constraints i edge cases pomagają ograniczyć błędy. Pseudokod oddziela logikę od składni, a weryfikacja powinna obejmować normalne i nietypowe przypadki.
