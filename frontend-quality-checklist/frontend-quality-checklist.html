Frontend Quality Checklist — HTML/CSS/JS przed pokazaniem projektu

Praktyczna karta przeglądu jakości frontendowej przed przekazaniem projektu klientowi, publikacją w portfolio lub wdrożeniem statycznej strony. Dokument wspiera szybki, techniczny przegląd: od semantyki i dostępności po deploy, README i higienę Git.

1. Wprowadzenie

Użyj tej checklisty jako końcowego przeglądu „przed pokazaniem”. Jej celem nie jest formalny audyt, tylko praktyczne wychwycenie ryzyk, które najczęściej psują odbiór projektu: niespójny kod, brak fallbacków, problemy z dostępnością, regressions na mobile i niedopilnowany proces publikacji.

Wypełniaj statusy bez „na oko”. Jeśli coś jest oznaczone jako OK, powinno być sprawdzone ręcznie lub potwierdzone w narzędziach.

4–16. Główna checklista jakości frontendowej

Checklista przeglądu jakości
Obszar Kryterium jakości OK Do poprawy Nie dotyczy Notatki / poprawki
4. Semantic HTML i struktura dokumentuPoprawna struktura HTML5 i brak błędnych zagnieżdżeń.
Jeden logiczny <main> na stronie.
Landmarki header/nav/main/section/footer użyte zgodnie z rolą treści.
Hierarchia nagłówków jest logiczna i wspiera czytanie oraz SEO.
Linki i przyciski są użyte zgodnie z przeznaczeniem interakcji.
Brak pustych lub dekoracyjnych elementów udających treść.
Atrybuty lang, title i meta są ustawione adekwatnie.
5. BEM i spójność nazewnictwaNazewnictwo klas jest konsekwentne i przewidywalne.
Brak przypadkowych nazw typu box1, test, new-style.
Podział blok / element / modyfikator jest czytelny.
Brak niepotrzebnego nadpisywania selektorów i „walki o specyficzność”.
Klasy opisują strukturę/komponent, a nie chwilowy wygląd.
Łatwo znaleźć powiązanie między HTML i CSS.
6. Architektura CSS i design tokensZmiennie CSS obejmują kolory, spacing, fonty i breakpointy.
Brak magic numbers bez uzasadnienia lub komentarza.
Spójny rytm odstępów i przewidywalny system spacingu.
Skala typografii jest jednolita i nieprzypadkowa.
Specyficzność selektorów pozostaje na kontrolowanym poziomie.
Usunięte nieużywane lub martwe reguły CSS.
Warstwy base/layout/components/utilities są rozdzielone tam, gdzie to ma sens.
7. Responsive design i stabilność layoutuZachowanie mobile-first jest utrzymane.
Brak poziomego scrolla na typowych szerokościach ekranu.
Odstępy i czytelność treści są dobre na mobile/tablet/desktop.
Breakpointy odpowiadają realnym potrzebom layoutu.
Media mają zarezerwowane wymiary lub aspect-ratio.
Nawigacja działa poprawnie na małych ekranach.
Karty i siatki nie łamią layoutu przy skrajnych treściach.
Brak CLS powodowanego przez obrazy, fonty lub późne przesunięcia.
8. JavaScript i progressive enhancementJavaScript nie blokuje podstawowego dostępu do treści.
Interakcje mają bezpieczne fallbacki tam, gdzie to potrzebne.
Brak błędów w konsoli podczas podstawowego przepływu.
Komponenty wspierają wiele instancji (jeśli dotyczy).
Brak sztywnej zależności od jednego ID bez uzasadnienia.
Event listenery są kontrolowane i nie dublują się.
Obsłużone scenariusze resize / Escape / click outside (gdzie potrzebne).
Kod jest modularny, czytelny i prosty w utrzymaniu.
9. Ruch, reduced motion i jakość interakcjiAnimacje są subtelne, celowe i wspierają UX.
prefers-reduced-motion jest obsłużone tam, gdzie występuje ruch.
Stany hover/focus/active są spójne i przewidywalne.
Brak transition: all na elementach interaktywnych.
Animacje nie powodują layout thrashing.
Mikrointerakcje nie przeszkadzają w korzystaniu ze strony.
10. Przegląd dostępności (a11y)Cała podstawowa ścieżka działa z klawiatury.
Focus states są widoczne i nieusuwane globalnie.
Linki mają sensowne teksty (bez „kliknij tutaj”).
Formularze mają poprawne powiązanie label z polami.
Komunikaty błędów/sukcesu są czytelne i dostępne.
Kontrast tekstu i elementów interaktywnych jest wystarczający.
Istotne obrazy mają adekwatny alt text.
ARIA jest używane tylko tam, gdzie semantyka HTML nie wystarcza.
Wykonany test ścieżki Tab / Shift+Tab / Enter / Escape.
11. Performance i LighthouseObrazy są zoptymalizowane rozmiarem i formatem.
Brak niepotrzebnie ciężkich assetów.
Strategia fontów jest świadoma (lokalne fonty lub uzasadnione zewnętrzne).
font-display ustawione tam, gdzie to potrzebne.
Lazy loading dla mediów niekrytycznych.
Ograniczone render-blocking resources.
Wykonana podstawowa weryfikacja Lighthouse.
Zespół zna wpływ LCP, CLS i INP na odbiór strony.
Brak zbędnych bibliotek do prostych funkcji.
12. SEO i metadaneUnikalny <title> dla strony.
Meta description jest uzupełniony i adekwatny.
Open Graph / social preview działa poprawnie.
Canonical URL ustawiony tam, gdzie ma sens.
Hierarchia nagłówków wspiera SEO i czytelność.
Linki wewnętrzne są opisowe i celowe.
Sitemap / robots są ustawione tam, gdzie wymagane.
noindex użyte wyłącznie intencjonalnie.
13. Formularze i walidacja (jeśli dotyczy)Pola mają poprawne typy i atrybuty.
Pola wymagane są oznaczone i uzasadnione.
Walidacja po stronie klienta działa i wspiera użytkownika.
Komunikaty błędów i sukcesu są jasne.
Honeypot lub podstawowa ochrona antyspamowa (gdzie potrzebna).
Submit ma stany disabled/loading i jasny feedback.
Przy formularzu jest notatka o prywatności danych.
Przetestowany realny przepływ formularza end-to-end.
14. Netlify / gotowość deploymentuFolder publikacji jest poprawnie wskazany.
Build command ustawiony tylko tam, gdzie faktycznie potrzebny.
Nie jest deployowana przypadkowa lub niepełna wersja projektu.
Redirecty działają poprawnie (jeśli dotyczy).
Strona 404 istnieje i nie prowadzi do ślepego końca (jeśli dotyczy).
Cache/security headers są ustawione tam, gdzie skonfigurowane.
Zmienne środowiskowe nie są ujawnione po stronie klienta.
Production URL przetestowany ręcznie po deployu.
Deployment nie generuje zbędnego „szumu” rebuildów.
15. README i dokumentacjaREADME jasno opisuje cel projektu.
Wskazany stack i główne technologie.
Struktura folderów jest opisana.
Komendy uruchomienia/build/test są aktualne (jeśli dotyczy).
Są notatki dotyczące deployu.
Opisane decyzje a11y, SEO i performance.
Known limitations są jawnie zapisane.
Opisane sensowne next steps.
Brak przestarzałych lub mylących instrukcji.
16. Higiena Git i handoff projektuWorking tree jest czysty przed przekazaniem projektu.
Commity są sensowne i opisują faktyczne zmiany.
Brak przypadkowych plików tymczasowych i artefaktów lokalnych.
Brak sekretów, tokenów i kluczy API w repo.
Nazwa brancha jest czytelna i adekwatna do zakresu pracy.
Opis pull requesta jest jasny i konkretny (jeśli używany).
Zmiany są możliwe do review i sensownie zgrupowane.
Finalny diff został ręcznie sprawdzony przed handoffem.
