# Calculation test evidence policy

## Current-behavior tests

`test/calculations.test.js` is a dependency-free Node.js regression harness for **current implementation behavior**. Its numeric expectations preserve outputs generated by the repository's existing formulas. They document behavior for the six direct models, reverse-search tolerance, employer cost, period shaping, options, and custom B2B ZUS inputs. They are **not tax-law reference cases** and passing them does not verify constants, formulas, eligibility, rounding, or 2026 accuracy.

When changing documented behavior without changing tax rules, label a test description and review note as “current model” or “current behavior.” Do not promote a value from this harness into an authoritative example.

## Source-confirmed accuracy tests (future)

Add an accuracy test only when all of the following are recorded:

1. An official source record exists in [`tax-source-register.md`](tax-source-register.md) with institution, source location, effective date, review date, and implementation mapping.
2. The expected case is traceable to that evidence (or a qualified specialist’s documented review where official guidance alone is insufficient).
3. The case states engagement facts, period, options, applicability, deductions, contributions, calculation order, and required rounding.
4. The test is clearly separated/named as `source-confirmed` and links to its source-register record.

Until then, use `UNVERIFIED`, `SOURCE REQUIRED`, and `REFERENCE CASE REQUIRED`; never call current outputs verified or authoritative.
